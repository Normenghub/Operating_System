
# 🔄 메모리 관리 (스와핑)

---

## 📌 스와핑 개요

### Demand Paging

필요한 페이지들만 디스크에서 꺼내와서 메모리에 탑재하는 기법. 동시에 실행되는 모든 프로그램의 페이지를 물리 메모리에 전부 올리면 공간 부족이 발생하기 때문에 Demand Paging이 필요하다.

메모리 계층은 **레지스터 → 캐시 → 메인 메모리 → Mass Storage** 순으로 계층적으로 구성되어 있으며, 위로 갈수록 속도가 빠르고 아래로 갈수록 용량이 커진다.

#### Demand Paging의 장점

- I/O 요구가 적음
- 메모리 공간 요구가 적음
- 빠른 응답
- 더 많은 유저(프로그램)를 동시에 수용 가능

#### SWAPPER (Lazy Swapper / Pager)

디스크에서 필요한 페이지를 메모리에 **Swap In**시키고, 필요 없는 페이지는 **Swap Out**시킨다. 단, **페이지가 실제로 필요하기 전까지는 절대 Swap In하지 않는다.**

## Valid-Invalid Bit

페이지가 현재 메모리에 있는지, 디스크에만 있는지를 구별하기 위해 사용한다.

| 상태 | 의미 |
|------|------|
| **Valid (v)** | 해당 페이지가 합법적이고 현재 메모리에 존재함 |
| **Invalid (i)** | 페이지가 유효하지 않거나, 유효하지만 현재 디스크에만 있음 |

- **Invalid reference** → 프로세스 중단 (abort)
- **Not-in-memory** → 디스크에서 Swap In (Page Fault 처리)

---

## ⚡ Page Fault

### Page Fault 처리 과정

1. **Trap 발생** (유효하지 않은 페이지 참조)
2. **Page Table 확인** → 잘못된 참조면 중단, 메모리에 없으면 다음 단계 진행
3. **Free frame 탐색**
4. **디스크에서 페이지를 가져와 Free frame에 탑재**
5. **Page Table 업데이트** → Valid bit를 `v`로 설정
6. **Page fault를 유발한 명령어 재시작 (Restart the instruction)**

### Demand Paging의 주요 활동 및 시간 비중

| 활동 | 소요 시간 |
|------|----------|
| 인터럽트 처리 | 수백 개의 instruction 필요 |
| 페이지 읽기 (Read page) | **가장 많은 시간 소요** |
| 프로세스 재시작 | 짧은 시간 소요 |

### 효율적인 접근 시간 (EAT)

```
EAT = (1 - p) × 메모리 접근 시간 + p × Page Fault Service Time
p = 0이면 page fault 없음, p = 1이면 매 참조마다 fault 발생
```

- **Page Fault Service Time** = page fault overhead + swap page out + swap page in + restart overhead
- `p = 0` → page fault 없음 (이상적)
- `p = 1` → 매 참조마다 fault 발생 (최악)

## 📋 Copy-on-Write (COW)

부모·자식 프로세스는 처음에 메모리 안에서 **동일한 페이지를 공유**한다. 둘 중 하나가 공유 페이지를 수정하면, 그 시점에 **해당 페이지만 복사**하여 각자 별도의 복사본을 가진다.

COW 덕분에 수정된 페이지만 복사되므로 **프로세스 생성이 효율적**이다.

---

## 💾 Swap Space

시스템 메모리가 부족할 때 보조기억장치(HDD/SSD)의 일부 공간을 메모리처럼 활용하는 영역.

- 물리 메모리의 **비활성 페이지**가 스왑 공간으로 이동 (Swap Out)
- 메모리 여유가 생기면 스왑 공간의 페이지가 물리 메모리로 복귀 (Swap In)
- 블록 크기는 **페이지 크기와 동일**
- **전용 파티션** 또는 **파일 시스템의 파일**로 존재 가능
- OS가 여러 프로세스에 대해 **큰 메모리 환상(illusion)** 을 지원할 수 있게 해줌

## 🔁 Swap Daemon / Page Daemon

OS가 메모리가 완전히 꽉 찰 때까지 기다렸다가 교체하는 것은 비효율적이므로, **백그라운드 스레드(Page Daemon)** 가 선제적으로 free memory를 확보한다.

| 기준선 | 동작 |
|--------|------|
| **LW (Low Watermark)** | 남은 메모리가 하한선 아래로 떨어지면 백그라운드 스레드 동작 시작 |
| **HW (High Watermark)** | free space가 상한선을 넘으면 백그라운드 스레드 동작 중단 |

## 🗂️ 어떤 타입의 페이지가 교체될까?

| 페이지 타입 | 처리 방식 |
|------------|----------|
| 커널 코드 | 스왑 안됨 |
| 커널 데이터 | 스왑 안됨 |
| 유저 프로세스의 페이지 테이블 | 스왑 안됨 |
| 유저 프로세스의 커널 스택 | 스왑 안됨 |
| 유저 코드 페이지 | Drop |
| 유저 데이터 페이지 | Drop 또는 Swap |
| 유저 힙/스택 페이지 | Swap |
| 유저 프로세스에 mmap된 파일 | Drop 또는 파일 시스템으로 |
| 페이지 캐시 페이지 | Drop 또는 파일 시스템으로 |

희생될 페이지는 **Page Replacement Policy**가 선정한다.

## 🔄 Page Replacement

### 필요성

Page fault가 발생했을 때 **빈 프레임이 없으면** 기존 페이지와 교체해야 한다.

### Dirty bit (Modify bit)

수정된 페이지만 디스크에 다시 기록하여 **페이지 전송 오버헤드를 줄이는** 비트.

### Page Replacement 과정

1. 원하는 페이지의 디스크 위치 탐색
2. 빈 프레임 탐색 → 있으면 사용, 없으면 **희생 프레임(victim frame)** 선택
3. 디스크에서 페이지를 가져와 프레임에 적재 후 Page Table 업데이트
4. Instruction 재시작

> ⚠️ Page fault 발생 시 최대 **2번의 페이지 전송**(Swap Out + Swap In)이 필요하여 EAT가 증가할 수 있다.

## 📊 페이지 교체 알고리즘

### FIFO (First-In, First-Out)

가장 오래된(먼저 들어온) 페이지를 교체.

- **Belady's Anomaly** 현상 존재: 프레임 수가 늘어날수록 오히려 page fault가 더 많아지는 역설적인 현상이 발생할 수 있음

### OPT (Optimal)

앞으로 **가장 오랫동안 사용되지 않을** 페이지를 교체.

- 최적의 성능이지만 미래를 알아야 하므로 **실제 구현 불가**
- 다른 알고리즘의 성능 비교 기준으로 활용

### LRU (Least Recently Used)

**가장 오랫동안 사용되지 않은** 페이지를 교체 (과거 기반 예측).

- 지역성(locality)이 있으면 OPT에 근사한 성능
- **Stack 알고리즘** 기반 → Belady's Anomaly 발생하지 않음
- 접근 빈도는 고려하지 않으며, 모든 워크로드에 최적은 아님

#### LRU 구현 방법 (H/W 지원 필요)

| 방법 | 설명 |
|------|------|
| **Counter 방식** | 페이지 테이블 엔트리마다 카운터를 두고, 페이지 참조 시 현재 클럭값을 복사 |
| **Stack 방식** | 페이지 참조 시 해당 페이지를 스택에서 꺼내 최상단에 삽입 |

### LRU 근사(Approximation) 알고리즘

진짜 LRU는 H/W 지원이 필요하고 느리기 때문에 근사 알고리즘을 사용한다.

#### ① Additional-Reference-Bits 알고리즘

- 각 페이지마다 **8비트** 추가
- 정기적인 인터벌(Regular Interval)마다 참조 비트를 최상위 비트로 shift → 나머지 비트는 오른쪽으로 1비트씩 이동 → 최하위 비트 버림
- 8번의 시간 인터벌 동안의 페이지 사용 이력을 숫자로 표현
- 예: `11000100`은 `01110111`보다 더 최근에 사용된 것

#### ② Second-Chance 알고리즘 (Clock 알고리즘)

**FIFO + H/W 제공 Reference bit** 조합.

- Reference bit = **1** → 두 번째 기회 부여, 비트를 0으로 초기화 후 통과
- Reference bit = **0** → 교체 대상으로 선정
- **Circular Queue of Pages** 구조로 구현
- 모든 비트가 1이면 → FIFO와 동일하게 동작

#### ③ Enhanced Second-Chance 알고리즘

**Reference bit + Modify bit** 조합으로 교체 우선순위 결정.

| Reference bit | Modify bit | 설명 |
|:---:|:---:|------|
| 0 | 0 | 최근 사용도, 수정도 안됨 → **교체 최우선** |
| 0 | 1 | 최근 미사용, 수정됨 → 교체 전 디스크 저장 필요 |
| 1 | 0 | 최근 사용, 수정 안됨 → 곧 다시 사용될 가능성 있음 |
| 1 | 1 | 최근 사용, 수정됨 → 교체 최후순위 |

교체 대상을 찾기 위해 Circular Queue를 여러 번 순회해야 할 수도 있다.

### LFU / MFU

| 알고리즘 | 설명 |
|---------|------|
| **LFU (Least Frequently Used)** | 참조 횟수가 가장 적은 페이지 교체. 활발히 사용되는 페이지는 큰 참조 횟수를 가질 것이라 가정 |
| **MFU (Most Frequently Used)** | 참조 횟수가 가장 많은 페이지 교체. 가장 적게 참조된 페이지는 이제 막 들어온 것이라 가정 |

## 📈 워크로드별 알고리즘 성능 비교

### 1. No-Locality Workload (랜덤 참조)

OPT를 제외한 나머지 알고리즘들은 거의 비슷한 성능을 보인다. 캐시 크기가 전체 워크로드를 담을 만큼 커지면 어떤 정책을 써도 **100% hit rate** 달성.

### 2. 80-20 Workload (지역성 있음)

- 80%의 참조가 20%의 페이지에 집중, 나머지 20%의 참조는 80%의 페이지를 랜덤하게 참조
- 캐시 크기가 커질수록 **OPT > LRU > FIFO, RAND** 순으로 성능 차이가 나타남
- LRU는 자주 사용되는 hot page를 메모리에 유지할 가능성이 높아 우수한 성능

### 3. Looping Sequential (연속 루프 참조)

- 50개 페이지를 순서대로 반복 참조하는 패턴
- 캐시 크기가 루프 페이지 수 N보다 작으면: **OPT > RAND > LRU/FIFO**
- LRU/FIFO는 이 워크로드에서 **취약** (직전에 사용한 페이지가 다음 루프에서 교체될 수 있음)
- 캐시 크기가 N 이상이 되면 모든 알고리즘이 동일하게 **100% hit rate** 달성

## 🌀 Thrashing

### 정의

**Page fault 비율이 너무 높아 CPU 이용률이 급격히 떨어지는 현상.**

프로세스에 충분한 프레임이 없으면 → page fault 발생 → 기존 프레임 교체 → 교체된 프레임을 다시 빠르게 필요로 함 → 반복 악순환.

### 연쇄 악화 과정

1. CPU 이용률 저하
2. OS는 다중 프로그래밍 정도를 높이려 함
3. 새로운 프로세스 추가 → 상황 더욱 악화

> Thrashing = 실행 중인 프로세스들의 메모리 수요가 물리 메모리를 초과(oversubscribed)하는 상태

### 발생 조건 및 해결

- **발생 조건**: 현재 locality의 크기 > 할당된 프레임 수
- **해결**: 일부 프로세스를 일시 중단 → 남은 프로세스들의 working set이 물리 메모리에 맞도록 조정
- local 또는 priority page replacement를 사용하여 영향 제한

---

## 🪟 Working Set 모델

지역성(Locality) 모델을 기반으로 **Thrashing을 예방**하는 모델.

| 용어 | 설명 |
|------|------|
| **Δ (Delta)** | 워킹 셋 윈도우 (고정된 페이지 참조 수, 예: 1000 instructions) |
| **WSᵢ** | 프로세스 Pᵢ의 워킹 셋 = 가장 최근 Δ 참조에서 사용된 페이지 집합 |
| **WSSᵢ** | WSᵢ의 크기 |
| **D** | 총 필요 프레임 수 = ΣWSSᵢ |

### Δ 크기에 따른 영향

- Δ가 **너무 작으면** → 전체 locality를 포함하지 못함
- Δ가 **너무 크면** → 여러 locality를 한꺼번에 포함
- Δ가 **∞이면** → 프로그램 전체를 포함

> **D > m (물리 메모리 프레임 수)** → Thrashing 발생 → 프로세스 하나를 중단

### 워킹 셋 추적 방법

- 타이머 인터럽트를 **5,000 참조마다** 발생
- 각 페이지마다 **2개의 비트**를 메모리에 유지
- 타이머 인터럽트 발생 시 현재 참조 비트값을 복사해 두고 모든 참조 비트를 **0으로 초기화**
- 두 비트 중 **하나라도 1이면** → 워킹 셋에 포함

## 📉 Page Fault Frequency (PFF)

Working Set이 과거를 통해 미래를 예측하는 방식이라면, **PFF는 page fault 발생 횟수를 직접 실시간으로 모니터링**하는 방식이다.

| 상황 | 조치 |
|------|------|
| page fault 비율이 **상한선(Upper bound)** 초과 | 프레임 **추가 할당** |
| page fault 비율이 **하한선(Lower bound)** 미만 | 프레임 **회수** |

시간이 지남에 따라 page fault 비율 그래프는 올라갔다 내려가는 **주기적인 경향**을 띤다.
