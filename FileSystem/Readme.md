# 🗂️ 파일 시스템 (File System)

---

## 📌 파일 시스템 개요

### 저장장치 추상화

파워가 꺼져도 데이터가 유지되어야 하므로 **하드디스크(HDD)** 나 **SSD** 같은 비휘발성 저장장치를 사용한다.

저장장치는 **512B 크기의 작은 방(섹터/블록)들이 0번부터 N-1번까지 일렬로 늘어선 배열**로 추상화된다.

#### 주요 Operation (Block Interface)

| 연산 | 설명 |
|------|------|
| `Identify()` | 전체 블록 수 N 반환 |
| `Read(start sector #, # of sectors)` | 시작 번호부터 지정 개수만큼 읽기 |
| `Write(start sector #, # of sectors)` | 시작 번호부터 데이터 기록 |

---

### 파일 (File)

- 파일은 **연속적인 바이트의 배열(linear array of bytes)**
- 고유 식별자인 **아이노드 번호(inode number)** 를 가짐
- OS는 파일의 내부 구조를 자세히 알지 못하고, 그냥 **바이트의 연속**으로 인식함 → 파일 내용의 의미는 애플리케이션이 해석
- 파일 타입: 텍스트, 소스코드, 오브젝트, 실행파일 등

#### 파일을 다룰 때 OS 안에서 세트로 움직이는 3가지

1. **파일 데이터 (File Data)**: 실제 내용물 (바이트 배열)
2. **파일 속성 / 메타데이터 (Inode)**: 파일 크기, owner, 접근 권한, 생성/수정 시간, 데이터 블록 위치 등
3. **파일 이름 (File Name)**: 루트부터의 전체 경로명(pathname)이 파일을 특정

#### POSIX Inode에 담기는 속성들

- 파일 타입 (regular, directory, symlink, …)
- 아이노드 번호
- 접근 권한 (rwx for owner / group / others)
- 하드 링크 수
- 소유자 User ID / Group ID
- 파일 크기 (bytes)
- 할당된 512B 블록 수
- 최근 접근 시간 (atime), 수정 시간 (mtime), 상태 변경 시간 (ctime)

---

### 디렉터리 (Directory)

- 디렉터리도 아이노드 번호를 가진 **특별한 파일**
- 하는 일: **텍스트 이름 ↔ 아이노드 번호** 를 짝지어 놓은 **Mapping table** (예: `foo: 10`)
- 계층형 트리 구조로 구성되어 파일 탐색 경로를 안내

---

### 파일 시스템 설계 목표

- **성능 (Performance)**
- **신뢰성 (Reliability)**
- **확장성 (Scalability)**

## 📂 파일 연산 (File Operations)

### POSIX 파일 시스템 인터페이스

| 함수 | 설명 |
|------|------|
| `open(pathname, flags, mode)` | 파일 열기 → file descriptor 반환 |
| `creat(pathname, mode)` | 파일 생성 |
| `read(fd, buf, count)` | read pointer 위치에서 읽기 |
| `write(fd, buf, count)` | write pointer 위치에 쓰기 |
| `lseek(fd, offset, whence)` | 파일 내 위치 이동 (seek) |
| `close(fd)` | 파일 닫기 |
| `fsync(fd)` | 메타데이터 + 데이터 모두 디스크에 강제 기록 |
| `fdatasync(fd)` | 데이터만 강제 기록 (메타데이터 제외) |
| `rename(oldpath, newpath)` | 파일 이름 변경 |
| `unlink(pathname)` | 이름 링크 제거 (reference count 감소) |
| `stat(path, buf)` | 파일 속성 조회 |
| `link(oldpath, newpath)` | 하드 링크 생성 |
| `symlink(oldpath, newpath)` | 심볼릭 링크 생성 |
| `mount(source, target, fstype, …)` | 파일 시스템 마운트 |
| `umount(target)` | 마운트 해제 |

### open()과 close()

파일을 읽고 쓸 때 가장 부하가 큰 작업 중 하나는 경로를 따라 디스크를 뒤져서 파일을 찾는 **검색 작업**이다. 이를 매번 반복하지 않도록 `open()` 시스템 콜을 먼저 호출해 파일 정보를 메모리에 올려둔다.

- **`open()`**: 디스크에서 디렉터리 구조를 탐색해 파일 항목을 메모리로 올리고, **file descriptor(정수)** 를 반환
- **`close()`**: 메모리에 있던 변경사항을 디렉터리 구조에 최종 반영

### 오픈 파일 테이블 (Open File Table)

`open()` 이 반환하는 file descriptor는 **프로세스별 파일 테이블**의 인덱스이며, 그 엔트리는 **시스템 전역 오픈 파일 테이블**의 항목을 가리킨다.

#### struct file (오픈 파일 테이블 엔트리)

| 필드 | 설명 |
|------|------|
| `int ref` | 현재 이 파일 엔트리를 가리키는 프로세스 수 (reference count) |
| `uint off` | 다음 읽기/쓰기 위치를 가리키는 포인터 (file position) |
| `readable / writable` | 읽기 전용인지, 쓰기 권한도 있는지 |
| `*ip` | 실제 아이노드를 가리키는 메모리 포인터 |

### Pathname Translation

`open("/a/b/c", …)` 호출 시 다음 단계를 거친다:

1. 루트 디렉터리 `/` 열기 (항상 위치 알 수 있음)
2. 디렉터리 항목에서 `a` 검색 → `a`의 위치 획득
3. 디렉터리 `a` 열기 → `b` 검색 → `b`의 위치 획득
4. 디렉터리 `b` 열기 → `c` 검색 → `c`의 위치 획득
5. 파일 `c` 열기

> **각 단계마다 권한(permission) 체크를 수행한다.**  
> 이 과정을 매번 반복하면 시간이 오래 걸리므로, OS는 **prefix lookup 결과를 캐시**에 올려 빠르게 재사용한다.

### Ensuring Persistence (영속성 보장)

파일을 쓸 때마다 디스크에 즉시 저장하면 너무 느리기 때문에, OS는 **페이지 캐시(page cache)** 라는 메모리 임시 저장소에 모아두었다가 최대 30초(Linux 기준) 후 디스크에 기록한다.

- **`fsync(fd)`**: 메타데이터와 모든 수정된 데이터를 디스크에 강제 flush
- **`fdatasync(fd)`**: 메타데이터를 제외하고 순수 데이터만 빠르게 flush

## 🔗 하드 링크 vs 심볼릭 링크

### 하드 링크 (Hard Link)

```bash
$ ln old.txt new.txt
```

- 두 파일이 **완전히 동일한 아이노드 번호를 공유** → 어느 쪽이 원본인지 구별 불가
- 아이노드는 **link count(reference count)** 를 유지하며, `unlink()` 호출 시 1 감소
- **link count가 0이 될 때** 아이노드와 데이터 블록이 실제로 삭제됨
- **제약**:
  - 서로 다른 파일 시스템 간 생성 불가
  - 디렉터리에 하드 링크 생성 불가

```bash
prompt> ls -i file file2
67158084 file   /* inode value is 67158084 */
67158084 file2  /* inode value is 67158084 */

prompt> rm file
removed 'file'
prompt> cat file2   // 여전히 접근 가능
hello
```

### 심볼릭 링크 (Symbolic / Soft Link)

```bash
$ ln -s old.txt new.txt
```

- Windows의 **바로가기 아이콘**과 같은 개념
- **경로명(문자열)을 자기 파일의 데이터 영역에 직접 저장**하는 별개의 새로운 파일
- 원본 파일이 삭제되어도 심볼릭 링크 파일 자체는 디스크에 살아있음 → 이 상태를 **dangling reference**라고 함

```bash
prompt> ln -s file file2
prompt> rm file          // 원본 삭제
prompt> cat file2
cat: file2: No such file or directory  // dangling reference
```

## 🔒 파일 보호 (File Protection)

### Access Control Lists (ACLs)

파일/디렉터리마다 모든 사용자와 허용 접근을 목록으로 관리하는 방식. 하지만 목록이 너무 커지는 문제가 있음.

### UNIX/Linux 해결책

대상과 행동을 분류하여 간결하게 표현:

| 구분 | 설명 |
|------|------|
| **대상** | Owner(파일 주인) / Group(같은 그룹) / Others(나머지) |
| **행동** | Read(4) / Write(2) / Execute(1) |

예시: `chmod 761 game`

- `7` (owner): 읽기+쓰기+실행 (4+2+1)
- `6` (group): 읽기+쓰기 (4+2)
- `1` (others): 실행만 (1)

## 💾 파일 시스템 마운트 (File System Mounting)

파일 시스템은 사용하기 전에 반드시 **마운트(mount)** 해야 한다.

| OS | 방식 |
|----|------|
| **Windows** | 드라이브 문자(C:, D:)를 부여해 독립 공간으로 관리 |
| **Unix/Linux** | 기존 디렉터리 트리 내 빈 폴더를 **마운트 포인트**로 지정해 붙임 |

Unix에서 `mount -t ext3 /dev/sda1 /home/users` 를 실행하면 `/home/users/` 경로가 새로 마운트된 파일 시스템의 루트를 가리킨다.

## 🗄️ 파일 시스템 구현 (File System Implementation)

### VSFS (Very Simple File System) 구조

디스크를 블록(예: 4KB) 단위로 나누고 다음 영역으로 구성:

| 영역 | 역할 |
|------|------|
| **Superblock (S)** | 파일 시스템 전체 요약 정보 (타입, 블록 크기, 총 블록 수, 아이노드 수 등). 마운트 시 가장 먼저 읽음 |
| **Inode bitmap (i)** | 각 아이노드가 사용 중(1)인지 빈(0)인지 표시 |
| **Data bitmap (d)** | 각 데이터 블록이 사용 중(1)인지 빈(0)인지 표시 |
| **Inode table (I)** | 파일 메타데이터 저장. 아이노드 하나는 보통 128~256B이므로 4KB 블록 하나에 16개 저장 가능 |
| **Data blocks (D)** | 실제 사용자 데이터 저장. 디스크 대부분의 공간을 차지 |

## 📦 할당 방법 (Allocation Methods)

파일의 논리적 데이터를 물리적 디스크 블록에 매핑하는 방법.  
고려 이슈: 외부 단편화, 파일 확장성, 순차/랜덤 접근 성능, 메타데이터 오버헤드.

### 1. 연속 할당 (Contiguous Allocation)

디렉터리에 **파일 이름 + start + length** 정보를 저장. 파일이 연속된 블록에 저장됨.

**장점**
- 메타데이터 오버헤드 거의 없음
- 최고의 순차 접근 성능
- 무작위 접근 계산이 단순 (`start + offset`)
- CD-ROM 등 읽기 전용 매체에 적합

**단점**
- **외부 단편화** 발생 (compaction 필요 → 비쌈)
- **동적 저장소 할당 문제** (first-fit / best-fit / worst-fit)
- 파일 생성 시 최종 크기를 미리 알아야 함 → 불편

### 2. 링크 할당 (Linked Allocation)

디렉터리에 **start + end** 저장. 각 블록 내부에 **다음 블록을 가리키는 포인터**를 저장하고, 마지막 블록은 -1.

**장점**
- 외부 단편화 완전 해결
- 파일 크기 자유롭게 확장 가능

**단점**
- **최악의 랜덤 접근 성능** (특정 위치로 이동하려면 처음부터 순회해야 함)
- 블록마다 포인터 저장으로 공간 낭비
- 블록 하나가 손상되면 이후 체인 전체 유실 → 신뢰성 낮음

#### FAT (File Allocation Table)

링크 할당의 단점을 보완. 테이블의 각 칸이 디스크 블록 번호와 1:1 매핑되고, 칸 안에 **다음 데이터 블록 번호** 기록. FAT를 메모리에 올려두어 포인터 탐색 속도를 높임 → 랜덤 접근 성능 향상.

### 3. 인덱스 할당 (Indexed Allocation)

디렉터리에 **파일 이름 + index block 번호** 저장. 인덱스 블록 안에 데이터 블록 위치 포인터들이 순서대로 정렬.

**장점**
- 외부 단편화 없음
- 랜덤 접근 뛰어남

**단점**
- 인덱스 블록 자체의 메타데이터 오버헤드
- 인덱스 블록이 너무 작으면 대용량 파일의 모든 블록 주소를 담을 수 없고, 너무 크면 내부 단편화 발생

#### 인덱스 블록 확장 방식

| 방식 | 설명 |
|------|------|
| **연결 스키마 (Linked scheme)** | 인덱스 블록이 가득 차면 다음 인덱스 블록을 가리키는 포인터로 체인 연결 |
| **다단계 스키마 (Multilevel scheme)** | 페이징의 멀티 레벨 페이지 테이블처럼 트리 구조로 확장 (예: 2-level → 최대 4GB) |
| **결합 스키마 (Combined scheme)** | 직접 포인터 12개 + single indirect + double indirect + triple indirect를 한꺼번에 지원 (Unix inode 방식) |

#### Unix inode (결합 스키마 예시)

- 12개 **direct 포인터** → 소용량 파일에 직접 접근
- 1개 **single indirect** → 중간 크기 파일
- 1개 **double indirect** → 대용량 파일
- 1개 **triple indirect** → 초대용량 파일

### 4. 익스텐트 기반 할당 (Extent-based Allocation)

여러 연속 블록 묶음(extent)을 B+ 트리 구조로 관리. 각 리프 노드에 **(시작 블록 번호, 연속 블록 개수)** 쌍을 기록. (예: Linux Ext4)

**장점**
- 메타데이터 오버헤드 대폭 감소
- 연속 접근 성능 유지
- 합리적 수준의 외부 단편화

## 🗃️ 빈 공간 관리 (Free Space Management)

| 방법 | 설명 |
|------|------|
| **비트 벡터 (Bit Vector)** | 블록당 1비트 (0=사용 중, 1=빈 공간). 단순하지만 디스크가 커질수록 비트맵 자체도 커짐 |
| **링크드 리스트 (Linked List)** | 빈 블록들을 링크드 리스트로 연결. 순회가 필요하지만 거의 발생하지 않음. FAT가 이 기법 활용 |
| **그룹핑 (Grouping)** | n개의 빈 블록 주소를 첫 번째 빈 블록에 묶어 저장, 그룹 간 포인터로 연결 |
| **카운팅 (Counting)** | 첫 번째 빈 블록 주소 + 연속으로 비어있는 블록 수를 하나의 쌍으로 기록 |

## ⚡ 파일 시스템 일관성 (File System Consistency)

### Crash Consistency 문제

하나의 시스템 콜(예: `write()`)을 완료하려면 여러 번의 디스크 쓰기가 필요하다. 디스크는 **단일 섹터 쓰기**만 원자성을 보장하므로, 중간에 크래시(전원 차단, 커널 패닉, 하드웨어 오류)가 발생하면 디스크 구조가 **불일치 상태**로 남을 수 있다.

#### 데이터 추가 시 변경되는 3가지 영역

파일에 새 데이터 블록(Db)을 추가하면:

1. **Data bitmap** 업데이트 (Db 블록을 사용 중으로 표시)
2. **Inode (I[v2])** 업데이트 (새 블록 포인터 + 파일 크기 증가)
3. **Data block(Db)** 실제 데이터 기록

#### 크래시 시나리오 7가지

| 시나리오 | 저장된 것 | 결과 | 문제 |
|---------|----------|------|------|
| 1 | 셋 다 저장됨 / 셋 다 저장 안 됨 | ✅ 문제 없음 | - |
| 2 | 데이터 블록만 | ✅ 안전 (장부 모순 없음) | - |
| 3 | 아이노드만 | ❌ 불일치 | 아이노드가 빈 블록을 가리킴 → 쓰레기값 읽힘, 동일 블록이 다른 파일에 할당될 수 있음 |
| 4 | 데이터 비트맵만 | ❌ 불일치 | 사용 중 표시됐지만 아무 아이노드도 가리키지 않음 → **공간 누수(space leak)** |
| 5 | 아이노드 + 비트맵 | ✅ 장부는 일치 (but 읽으면 쓰레기값) | - |
| 6 | 아이노드 + 데이터 블록 | ❌ 불일치 | 비트맵이 블록을 빈 것으로 보아 → 다른 파일에 덮어쓸 수 있음 |
| 7 | 비트맵 + 데이터 블록 | ❌ 불일치 | 아이노드 없음 → 해당 블록에 영원히 접근 불가, **공간 낭비** |

## 해결책 1: FSCK (File System Checker)

크래시 후 재부팅 시 **디스크 전체를 전수 조사**해 모순을 강제로 수정하는 고전 Unix 도구 (Windows의 Scandisk에 해당).

### 검사 및 수정 항목

- 아이노드 및 데이터 **비트맵 일관성** 검사 → 불일치 시 비트맵 업데이트
- **중복 포인터** 발견 시 복사본을 새 블록에 저장
- **아이노드 link count** 수정
- 디렉터리를 찾을 수 없는 파일 → `/lost+found` 임시 디렉터리에 생성

### FSCK의 문제점

- **너무 느림**: 5단계를 거쳐 수억 개의 블록 포인터를 모두 검사 → 600GB 디스크 기준 약 70분 소요
- 크래시 직전 사용자가 **정확히 어떤 내용을 쓰려 했는지** 알 수 없음 (올바른 상태가 아닌 일관된 상태만 복원)

## 해결책 2: 저널링 (Journaling)

데이터베이스의 **write-ahead logging** 기법을 파일 시스템에 적용. 현대 파일 시스템(Linux Ext3/4, NTFS 등)에서 표준으로 사용.

### 동작 원리

디스크 내부에 **저널링 구역(journaling area)** 을 만들고:

1. **Journal write**: 헤더 블록(TxB) + 변경할 아이노드/비트맵/데이터 블록의 복사본을 저널에 먼저 기록
2. **Journal commit**: 커밋 블록(TxE) 기록 → 트랜잭션 확정
3. **Checkpoint**: 저널 내용을 실제 디스크 위치에 반영

### 저널링 기반 크래시 복구

| 크래시 시점 | 복구 방법 |
|------------|----------|
| Step 1~2 사이 (커밋 전) | 저널 폐기 → 이전 상태로 롤백 |
| Step 2~3 사이 (커밋 후, 체크포인트 전) | 저널 데이터로 **Redo logging** (재반영) |

### 저널링 최적화 기법

| 기법 | 설명 |
|------|------|
| **순환 로그 (Circular log)** | 저널 공간이 유한하므로 완료된 트랜잭션을 지우고 원형으로 재사용 |
| **로그 업데이트 묶기 (Batching)** | 여러 쓰기 연산을 메모리에 모았다가 주기적으로 한꺼번에 기록 (Ext3/4는 약 5초) |
| **저널 체크섬 (Journal checksum)** | 저널 쓰기와 커밋 사이의 write barrier를 제거해 성능 향상 |
| **메타데이터 저널링 (Metadata journaling)** | 데이터 블록은 저널에 기록하지 않고 메타데이터만 저널링 → 성능↑. Ext3/4의 **Ordered journaling**: 데이터 블록을 먼저 쓴 뒤 저널을 커밋해 쓰레기값 방지 |
